# Policy and Contact Pages Design

**Date:** 2026-05-24
**Status:** Approved
**Scope:** Add original Azani static policy and contact pages inspired by common baby-shop coverage: shipping, returns, privacy, terms, and contact.

## Goals

1. Replace missing or disabled customer-service destinations with real pages.
2. Give shoppers practical answers before checkout: delivery timing, return eligibility, privacy basics, payment/order terms, and support channels.
3. Keep the implementation simple, testable, and consistent with the current Azani storefront design system.

## Non-goals

- Copying third-party policy text.
- Attorney-reviewed legal terms.
- A CMS or admin editor for static content.
- A full navigation redesign. The current Kiabi-leaning header remains primary.

## Content

The new pages are:

- `/policies/shipping`
- `/policies/returns`
- `/policies/privacy`
- `/policies/terms`
- `/contact`

Each policy page includes a title, short summary, "Last updated: May 24, 2026", quick facts, and scannable sections. Copy is Azani-specific and uses the existing contact defaults in `src/lib/site-config.ts`: Nairobi, Kenya; `hello@azani.shop`; `+254 700 000 000`; WhatsApp support.

The content should mention:

- Nairobi and Kenya delivery coverage.
- Free delivery threshold already used in the UI: KSh5,000.
- Standard and express delivery language already used in checkout.
- Returns within 3 days for unused items in original packaging.
- Damaged or incorrect item reporting within 48 hours.
- Hygiene-sensitive exclusions for opened feeding, bath, diapering, and personal-care products.
- M-Pesa, card, and mobile-money payment language.

## Architecture

Create a single content module, `src/lib/store-info-pages.ts`, containing typed page data. Create a reusable renderer, `src/components/store-info-page.tsx`, for the visual layout. Individual App Router page files import one page object and render it.

This avoids copy-pasted JSX while keeping routes explicit and statically generated by Next.js.

## Footer

Update `src/components/site-footer.tsx` so the disabled customer-service text becomes real links:

- Checkout -> `/checkout`
- Shipping Policy -> `/policies/shipping`
- Returns & Exchanges -> `/policies/returns`
- Privacy Policy -> `/policies/privacy`
- Terms of Service -> `/policies/terms`
- Contact -> `/contact`

The footer should keep its current four-column layout.

## Testing

Add tests before implementation:

- Static page tests render each new route and assert its heading and key Azani-specific content.
- Footer tests assert the new policy links are clickable and target the expected routes.
- Content tests assert the policy registry contains all expected routes and does not contain third-party brand/source names.

Run focused tests first, then full verification.

## Risks

- Policy copy may need later legal review. Mitigation: keep language practical and store-operational rather than overclaiming.
- Real delivery fees may differ by order. Mitigation: state that exact rates are confirmed at checkout.
